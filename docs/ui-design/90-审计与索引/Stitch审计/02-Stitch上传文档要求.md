# Stitch Upload Document Requirements

## Purpose

This document defines the document taxonomy, minimum required fields, authority
levels, upload rules, usability gate, and conflict resolution order for Reader
for Android Stitch source preparation.

It defines requirements only. It does not classify, approve, reject, or audit
the current 163 candidate files.

## Common Rules

- Every candidate must be assigned exactly one category before upload review.
- A field is required only when it has an explicit, reviewable value. A heading
  or placeholder without a value does not satisfy a required field.
- `humanReviewResult` uses `PENDING`, `PASS`, or `FAIL`.
- `usable` defaults to `false`.
- `referenceOnly` material may provide context but must not override design
  authority.
- `provenanceOnly` material may identify files and versions but must not define
  visual or interaction behavior.
- `UNKNOWN` material must be classified before it can be uploaded or used.

## Document Taxonomy

| Category | Purpose | Default authority |
| --- | --- | --- |
| `SOURCE_UI_IMAGE` | Human-produced or human-selected source UI image for a mapped screen or state. | `DESIGN_AUTHORITY` only after human approval; otherwise no usable authority |
| `PAGE_SPEC` | Page goal, structure, states, navigation, and viewport definition. | `SUPPORTING_SPEC` |
| `INFORMATION_ARCHITECTURE` | Navigation model, page hierarchy, entry points, and exclusions. | `SUPPORTING_SPEC` |
| `DESIGN_SYSTEM` | Cross-page visual language and semantic design rules. | `SUPPORTING_SPEC` |
| `TOKEN_SPEC` | Named visual tokens and scales used by pages and components. | `SUPPORTING_SPEC` |
| `COMPONENT_SPEC` | Reusable component anatomy, variants, states, and usage rules. | `SUPPORTING_SPEC` |
| `INTERACTION_SPEC` | Triggers, flows, transitions, gestures, and non-happy-path behavior. | `SUPPORTING_SPEC` |
| `ANDROID_IMPL_NOTE` | Android platform constraints and implementation suggestions. | `IMPLEMENTATION_REFERENCE` |
| `AUDIT_DOC` | Point-in-time audit findings and severity/status records. | `AUDIT_REFERENCE` |
| `FIX_QUEUE` | Tracked issues and required fixes. | `AUDIT_REFERENCE` |
| `MANIFEST_OR_HASH` | File identity, path, hash, size, classification, and review provenance. | `FILE_PROVENANCE_ONLY` |
| `GENERATED_STITCH_OUTPUT` | Output generated by Stitch from supplied source material. | `GENERATED_NOT_AUTHORITY` |
| `HISTORY_OR_DEPRECATED` | Superseded, rejected, archived, or historical material. | `DEPRECATED_NOT_AUTHORITY` |
| `UNKNOWN` | Material not yet classified. | No authority |

## Required Fields

### `SOURCE_UI_IMAGE`

- `screenName`
- `theme`
- `sourcePath`
- `sourceVersion` or `capturedAt`
- `humanApproved`
- `targetScreenMapping`
- `usable` with default value `false`

- `pageGoal`
- `layoutStructure`
- `primaryRegions`

- `pageHierarchy`
- `entryPoints`

- `platformConstraint`

### `AUDIT_DOC`

- `auditTarget`
- `auditDate`
- `auditResult`
- `P0/P1/P2`

### `FIX_QUEUE`

- `issueId`
- `affectedScreen`
- `priority`
- `requiredFix`
- `status`
- `reviewResult`

### `MANIFEST_OR_HASH`

- `fileName`
- `sourcePath`
- `fileType`
- `sha256`
- `size`
- `category`
- `uploadStatus`
- `humanReviewResult`
- `usable`

### `GENERATED_STITCH_OUTPUT`

- `projectId`
- `generatedAt`
- `promptSource`
- `sourceMapping`
- `humanReviewResult`
- `usable` with default value `false`

### `HISTORY_OR_DEPRECATED`

- `reasonDeprecated`
- `replacedBy`
- `doNotUseAsAuthority: true`

### `UNKNOWN`

`UNKNOWN` has no authority-bearing required fields. It must record enough file
identity to be tracked, but it must be reclassified before upload review:

- `fileName`
- `sourcePath`
- `classificationStatus: PENDING`
- `usable: false`

## Authority Levels

| Authority level | Meaning |
| --- | --- |
| `DESIGN_AUTHORITY` | May determine visual design when the applicable human approval gate has passed. |
| `SUPPORTING_SPEC` | May define structure, tokens, components, or interaction details without overriding higher authority. |
| `IMPLEMENTATION_REFERENCE` | May guide Android implementation but must not change approved design. |
| `AUDIT_REFERENCE` | May report findings or work status but is not a design source. |
| `FILE_PROVENANCE_ONLY` | May prove file identity and history only. |
| `GENERATED_NOT_AUTHORITY` | Generated output is never source authority by itself. |
| `DEPRECATED_NOT_AUTHORITY` | Must not be used as current authority. |

## Upload Allowed Rules

| Category | Upload allowed | Upload role and conditions |
| --- | --- | --- |
| `SOURCE_UI_IMAGE` | Yes | Source candidate; `usable=false` by default and human approval is required before use. |
| `PAGE_SPEC` | Yes | Supporting source specification. |
| `INFORMATION_ARCHITECTURE` | Yes | Supporting source specification. |
| `DESIGN_SYSTEM` | Yes | Supporting source specification. |
| `TOKEN_SPEC` | Yes | Supporting source specification. |
| `COMPONENT_SPEC` | Yes | Supporting source specification. |
| `INTERACTION_SPEC` | Yes | Supporting source specification. |
| `ANDROID_IMPL_NOTE` | Yes | `referenceOnly`; must retain `mustNotOverrideDesign: true`. |
| `AUDIT_DOC` | Yes | `referenceOnly`. |
| `FIX_QUEUE` | Yes | `referenceOnly`. |
| `MANIFEST_OR_HASH` | Yes | `provenanceOnly`. |
| `GENERATED_STITCH_OUTPUT` | No by default | Must not be uploaded as source. A separate explicit human decision is required for any exception, and it remains non-authoritative. |
| `HISTORY_OR_DEPRECATED` | Reference only or exclude | Must never be uploaded as design authority. |
| `UNKNOWN` | No | Classify first. |

Upload permission does not grant authority and does not set `usable=true`.

## Usable Rules

- Generated, existing, or uploaded does not mean usable.
- A complete manifest entry does not mean usable.
- A model audit result of `PASS` does not mean usable.
- `humanApproved=true` without `humanReviewResult=PASS` does not set
  `usable=true`.
- Only `humanReviewResult=PASS` permits `usable=true`.
- `humanReviewResult=PENDING` or `humanReviewResult=FAIL` requires
  `usable=false`.
- Human review must identify the reviewed version or hash and the applicable
  target screen mapping.
- `AUDIT_DOC`, `FIX_QUEUE`, `MANIFEST_OR_HASH`, `GENERATED_STITCH_OUTPUT`,
  `HISTORY_OR_DEPRECATED`, and `UNKNOWN` cannot become design authority merely
  by setting `usable=true`.

## Conflict Resolution Order

When two sources conflict, apply the first applicable source in this order:

1. Human-approved `SOURCE_UI_IMAGE`
2. `STITCH_DESIGN_SOURCE_OF_TRUTH.md`
3. `PAGE_SPEC` / `INFORMATION_ARCHITECTURE`
4. `DESIGN_SYSTEM` / `TOKEN_SPEC` / `COMPONENT_SPEC`
5. `INTERACTION_SPEC`
6. `ANDROID_IMPL_NOTE`
7. `AUDIT_DOC` / `FIX_QUEUE`
8. `MANIFEST_OR_HASH`
9. `GENERATED_STITCH_OUTPUT`
10. `HISTORY_OR_DEPRECATED`

Additional rules:

- A lower-ranked source must not override a higher-ranked source.
- A lower-ranked source may fill an unspecified detail only when it does not
  contradict higher-ranked material.
- `UNKNOWN` does not participate in conflict resolution.
- If equally ranked sources conflict and neither has an explicit later
  human-approved version, keep `usable=false` and require human resolution.

## Stitch No-Redesign Rule

Stitch must reproduce the supplied, approved design intent. It must not:

- invent a new visual direction;
- replace approved layout, hierarchy, component anatomy, tokens, or navigation;
- treat generated output as a new source of truth;
- resolve source conflicts by aesthetic preference;
- fill an unspecified design decision beyond the available authority chain.

An unresolved design decision must remain unresolved until human review supplies
or approves an authoritative source.
